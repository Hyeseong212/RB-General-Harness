# 고급 하네스 설계 원칙

단순한 하네스는 "이렇게 답해줘"라고 말합니다. 고급 하네스는 "이 입력을 받아서, 이 역할로, 이 절차를 거쳐, 이 검증을 통과한 산출물을 남겨라"라고 정합니다.

## 1. 규칙은 짧고 적용 범위는 좁게 둔다

나쁜 예:

```md
항상 최고의 품질로 작업하고, 최신 기술을 사용하고, 깔끔하게 작성한다.
```

좋은 예:

```md
React 화면을 수정할 때는 기존 컴포넌트 구조를 먼저 확인하고, 새 디자인 시스템을 만들지 않는다.
수정 후 데스크톱과 모바일 캡처를 남긴다.
```

## 2. 부정문보다 행동 동사를 쓴다

나쁜 예:

```md
대충 하지 마라.
```

좋은 예:

```md
결과를 제출하기 전에 입력 자료, 변경 내용, 검증 결과, 남은 리스크를 각각 3줄 이내로 요약한다.
```

## 3. 입력 자료를 명시한다

에이전트가 모르는 자료를 추측하지 않게 합니다.

```md
## 입력 자료
- 회의록: docs/meeting/2026-05-11.md
- 기존 PPT: 발표자료/v2.pptx
- 캡처 이미지: captures/*.png
- 참고 금지: 아직 승인되지 않은 견적서
```

## 4. 산출물 스키마를 고정한다

결과 형식이 매번 바뀌면 팀 운영에 쓰기 어렵습니다.

```md
## 산출물
1. 요약
2. 변경 내용
3. 검증 결과
4. WARN 항목
5. 다음 액션
```

## 5. 사람 승인 게이트를 문서 안에 둔다

업무상 확정하면 안 되는 영역을 파일 안에 명시합니다.

```md
## 사람 승인 필요
- 환자 정보가 포함된 화면 캡처
- 비용 집행 또는 계약 조건
- 의료적 효과를 단정하는 문장
- 운영 DB 변경
```

## 6. 메모리와 인계를 분리한다

규칙 파일은 자주 바꾸지 않고, 작업 상태는 별도 메모리 파일에 씁니다.

- 규칙: 거의 변하지 않는 운영 원칙
- 메모리: 최근 결정, 진행 상태, 남은 이슈
- 인계: 이번 작업의 결과와 다음 액션

## 7. 하네스는 단계별로 커진다

처음부터 복잡하게 만들 필요는 없습니다.

1. 반복되는 질문을 MD로 만든다.
2. 산출물 형식을 고정한다.
3. 검증 기준을 추가한다.
4. 인계 문서를 추가한다.
5. 사람 승인 게이트를 추가한다.
6. 필요하면 자동 검증 또는 subagent를 붙인다.

## 하네스 품질 체크리스트

- 이 문서만 보고 새 사람이 작업을 맡길 수 있는가
- 에이전트가 무엇을 추측하면 안 되는지 적혀 있는가
- 결과물 파일 위치가 정해져 있는가
- PASS, WARN, FAIL 기준이 있는가
- 사람 승인 없이는 확정하면 안 되는 항목이 표시되어 있는가
- 다음 회차 작업자가 이어받을 수 있는 인계 형식이 있는가
