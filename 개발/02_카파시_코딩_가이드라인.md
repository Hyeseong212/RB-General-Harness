# 카파시 코딩 가이드라인

이 문서는 LLM 코딩 에이전트가 자주 만드는 실수를 줄이기 위한 개발 원칙이다.  
Andrej Karpathy가 언급한 LLM 코딩 실수 관찰을 바탕으로 정리한다.

## 1. 코딩 전에 생각한다

가정하지 말고, 애매한 부분을 숨기지 않는다.

구현 전에 확인할 것:

- 내가 이해한 요구사항을 명시한다.
- 여러 해석이 가능하면 조용히 하나를 고르지 않는다.
- 더 단순한 방법이 있으면 먼저 말한다.
- 불명확하면 멈추고 질문한다.

## 2. 단순함을 우선한다

요구사항을 해결하는 최소 코드만 작성한다.

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰는 코드를 위해 추상화를 만들지 않는다.
- 요구되지 않은 설정 가능성이나 확장성을 넣지 않는다.
- 실제로 발생하지 않는 상황을 위한 과한 예외 처리를 넣지 않는다.
- 200줄로 썼는데 50줄로 해결 가능하면 다시 줄인다.

질문:

> 시니어 개발자가 이 코드를 보고 과하다고 말할까?

그렇다면 단순화한다.

## 3. 필요한 부분만 고친다

기존 코드를 수정할 때는 요청과 직접 관련된 부분만 건드린다.

- 주변 코드, 주석, 포맷을 임의로 개선하지 않는다.
- 고장 나지 않은 코드를 리팩토링하지 않는다.
- 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 삭제하지 말고 언급만 한다.

내 변경으로 생긴 불필요한 import, 변수, 함수는 제거한다.  
기존부터 있던 죽은 코드는 요청받지 않는 한 제거하지 않는다.

기준:

> 모든 변경 라인은 사용자의 요청과 직접 연결되어야 한다.

## 4. 검증 가능한 목표로 실행한다

작업을 검증 가능한 목표로 바꾼다.

예시:

- "검증 추가" -> "잘못된 입력 테스트를 만들고 통과시킨다"
- "버그 수정" -> "버그를 재현하는 테스트를 먼저 만들고 통과시킨다"
- "리팩토링" -> "수정 전후 테스트가 모두 통과하게 한다"

여러 단계 작업은 짧은 계획을 세운다.

```text
1. [단계] -> 검증: [확인 방법]
2. [단계] -> 검증: [확인 방법]
3. [단계] -> 검증: [확인 방법]
```

좋은 완료 기준은 에이전트가 독립적으로 반복 검증할 수 있게 한다.  
"잘 되게 해줘" 같은 약한 기준은 계속 확인 질문을 만들게 된다.
