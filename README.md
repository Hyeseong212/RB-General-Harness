# Agent Harnessing Guidelines

이 저장소는 회사 내부에서 AI 에이전트를 개인 프롬프트가 아니라 반복 가능한 업무 산출물 구조로 쓰기 위한 가이드라인 모음입니다.

좋은 하네스는 "AI에게 이렇게 답해줘" 수준에서 끝나지 않습니다. 입력 자료, 적용 범위, 역할, 작업 절차, 검증 기준, 산출물 위치, 사람 승인 게이트를 파일로 고정해서 다음 작업에서도 재사용할 수 있어야 합니다.

## 하네스 레벨

### Level 1. 기본 규칙

작업할 때 지켜야 할 공통 원칙입니다. 예: "민감 정보는 마스킹한다", "검증 결과를 PASS/WARN/FAIL로 남긴다".

### Level 2. 작업 계약서

특정 업무를 맡길 때 입력, 절차, 산출물, 검증 기준을 고정한 문서입니다. 예: 개발 Task brief, PPT 캡처 정리, 구매 견적 비교.

### Level 3. 메모리와 인계

세션이 바뀌어도 이어갈 수 있도록 프로젝트 상태, 최근 결정, 남은 리스크, 다음 액션을 따로 남깁니다.

### Level 4. 검증 게이트

사람 승인, 테스트, 리뷰, 보안 확인, 대외 제출 확인처럼 결과가 실제 업무로 넘어가기 전의 관문입니다.

### Level 5. 팩트체킹 게이트

법률, 규제, 가격, 정부지원사업, 의료기기 인허가처럼 최신성이 중요한 내용은 모델 기억으로 답하지 않고 공식 출처를 실제 확인합니다. 확인 날짜, 출처 URL, 근거 문장, 사람 승인 필요 여부를 남깁니다.

## 기본 사용 순서

1. `templates/advanced-task-brief.md`로 작업 요청을 작성한다.
2. 부서별 README에서 해당 작업의 검증 기준을 확인한다.
3. 에이전트에게 입력 자료와 산출물 위치를 명확히 준다.
4. 결과를 `templates/handoff-note.md` 형식으로 인계한다.
5. 사람 승인 게이트가 필요한 항목은 담당자가 확정한다.

## 참고한 공개 패턴

- `AGENTS.md`: 에이전트가 읽는 저장소용 README 패턴
- Cursor Rules: 프로젝트와 파일 범위별로 규칙을 분리하는 방식
- Cline Memory Bank: 세션 간 프로젝트 맥락을 유지하는 문서 구조
- Claude Code Subagents/Hooks: 역할 분리와 자동 검증 게이트 구조
- Aider CONVENTIONS.md: 코딩 규칙을 읽기 전용 컨텍스트로 주입하는 방식
- GitHub Copilot instructions/prompt files: 저장소 공통 지침과 작업별 프롬프트 파일 분리
- 국가법령정보센터, 식품의약품안전처, 개인정보보호위원회 등 공식 출처: 법률·규제·의료기기 관련 최신성 검증
