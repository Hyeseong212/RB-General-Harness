# 개발 에이전트 운영원칙

## 목적

개발 에이전트는 개발자를 대체하는 역할이 아니라, 분석, 구현, 검증, 문서화를 Task 단위로 빠르게 묶는 역할이다.

개발자는 최종 판단자이고, 에이전트는 작업 속도를 높이는 실행 보조자다.

## 기본 구조

개발 업무는 한 에이전트에게 모두 맡기지 않고 역할을 나눈다.

| 역할 | 담당 |
|---|---|
| 전체관리자 | 목표, 범위, 우선순위, 산출물 통합 |
| 레거시 분석 | 기존 코드, DB, API, 화면 동작 조사 |
| 구현 | 정해진 파일 범위 안에서 코드 변경 |
| 코드리뷰 | 버그, 회귀, 보안, 누락 테스트 확인 |
| QA | 빌드, API, UI, 권한, 데이터 범위 검증 |
| 문서화 | 변경 내용, 리스크, 다음 작업 기록 |

## 개발 에이전트에 맡기기 좋은 일

- 레거시 코드 구조 분석
- API 요청/응답 정리
- 화면과 백엔드 연결 흐름 정리
- 반복 CRUD 화면 구현
- 테스트 케이스 초안 작성
- 빌드 오류 원인 분석
- QA 로그 요약
- 변경 내용 문서화

## 개발자가 직접 판단해야 하는 일

- 아키텍처 최종 결정
- 운영 DB 변경
- 보안 정책 변경
- 개인정보/민감정보 처리 방식
- 배포 여부
- 장애 대응 우선순위

## 작업 전 입력해야 할 것

- 목표 기능
- 관련 파일 또는 폴더
- 기준이 되는 기존 동작
- 수정 가능한 범위
- 건드리면 안 되는 범위
- 완료 기준
- 테스트 방법

## 완료 기준

개발 Task는 아래 중 가능한 항목을 통과해야 완료로 본다.

- 빌드 통과
- 주요 API 응답 확인
- 화면 동작 확인
- 권한별 접근 확인
- 기존 기능 회귀 없음
- 변경 내용과 남은 리스크 문서화

## 금지 패턴

- "전체적으로 고쳐줘"처럼 범위가 넓은 요청
- 레거시 기준 확인 없이 새 화면만 만드는 방식
- 테스트 없이 코드만 받는 방식
- 관련 없는 파일까지 리팩토링하는 방식
- 에이전트가 만든 코드를 사람이 읽지 않고 바로 반영하는 방식
