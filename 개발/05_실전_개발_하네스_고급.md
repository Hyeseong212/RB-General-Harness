# 실전 개발 하네스 고급

개발 하네스는 "코드 작성 규칙"만으로는 부족합니다. 실제 운영에서는 탐색, 구현, 테스트, 리뷰, 인계가 분리되어야 합니다.

## 개발 Task 분해

| 단계 | 역할 | 산출물 |
| --- | --- | --- |
| 탐색 | Explorer | 관련 파일, 기존 패턴, 변경 후보 |
| 계획 | Planner | 작업 범위, 위험, 검증 방법 |
| 구현 | Implementer | 코드 변경, 마이그레이션, 테스트 |
| 검증 | Reviewer | 버그 가능성, 테스트 누락, 회귀 위험 |
| 인계 | Documenter | 변경 요약, 실행 명령, 남은 리스크 |

## Explorer 하네스

```md
목표: 구현하지 말고 관련 파일과 기존 패턴만 찾아라.
출력:
- 관련 파일
- 현재 흐름
- 변경이 필요한 지점
- 건드리면 위험한 지점
- 구현자에게 넘길 요약
금지:
- 코드 수정
- 추측으로 결론 확정
```

## Implementer 하네스

```md
목표: 지정된 파일 범위 안에서만 변경한다.
입력:
- Explorer 요약
- Task brief
- 기존 테스트 명령
출력:
- 변경 파일
- 변경 이유
- 실행한 검증
- 실행하지 못한 검증
금지:
- 요청 밖 리팩토링
- 관련 없는 포맷팅
- 검증 없이 완료 선언
```

## Reviewer 하네스

```md
목표: 칭찬보다 결함, 회귀, 누락 테스트를 먼저 찾는다.
출력:
- 치명도별 문제
- 파일/라인 근거
- 재현 가능성
- 누락된 검증
- 승인 또는 수정 요청
금지:
- 취향 기반 지적
- 근거 없는 추측
```

## 개발 검증 게이트

| 항목 | PASS | WARN | FAIL |
| --- | --- | --- | --- |
| 범위 | 요청 범위 안에서만 수정 | 주변 파일 일부 영향 | 불필요한 대규모 변경 |
| 테스트 | 관련 테스트 통과 | 테스트 일부 미실행 | 검증 없음 |
| 스타일 | 기존 패턴 유지 | 일부 새 패턴 | 프로젝트 스타일과 충돌 |
| 인계 | 변경 이유와 실행 명령 명확 | 리스크 일부 남음 | 다음 담당자가 이어받기 어려움 |

## 서브 에이전트 생성 기준

서브 에이전트는 "똑똑해 보이게 많이 나누는 것"이 아니라, 메인 작업의 컨텍스트를 보호하기 위해 씁니다.

사용하면 좋은 경우:

- 탐색과 구현이 분리될 때
- 프론트엔드, 백엔드, DB처럼 파일 소유 범위가 분리될 때
- 리뷰를 독립적으로 돌려야 할 때
- 테스트 실패 원인을 별도 가설로 조사할 때

피해야 하는 경우:

- 같은 파일을 여러 명이 동시에 고칠 때
- 결정을 계속 왕복해야 할 때
- 요구사항이 아직 모호할 때
- 단순 수정인데 오히려 인계 비용이 커질 때
