# 서브에이전트 생성 기준

## 목적

서브에이전트는 일을 많이 시키기 위한 장치가 아니라, 컨텍스트를 나누고 책임 범위를 분리하기 위한 장치다.

잘 나눈 서브에이전트는 병렬 작업을 가능하게 하지만, 잘못 나누면 결과 통합 비용이 더 커진다.

## 생성하면 좋은 경우

### 1. 컨텍스트가 큰 경우

아래 상황에서는 서브에이전트가 유리하다.

- 레거시 코드가 크고 파일 수가 많다.
- DB, API, 프론트엔드, Unity 같은 영역이 동시에 얽혀 있다.
- 하나의 대화 안에서 모든 자료를 읽으면 맥락이 흐려질 가능성이 크다.

예시:

- `레거시 분석 에이전트`: 기존 OFBiz 화면과 서비스 흐름만 조사
- `백엔드 에이전트`: Spring Boot API 변경만 담당
- `프론트엔드 에이전트`: Vue 화면과 라우팅만 담당
- `QA 에이전트`: 구현 후 테스트와 리포트만 담당

### 2. 모듈이 명확히 나뉘는 경우

서로 다른 파일 또는 책임 범위가 분리되어 있으면 병렬 작업이 가능하다.

좋은 예:

- 프론트엔드 화면 A 수정
- 백엔드 API B 수정
- QA 시나리오 C 작성

나쁜 예:

- 같은 파일을 여러 에이전트가 동시에 수정
- 같은 함수의 설계를 여러 에이전트에게 따로 맡김
- 최종 정책 판단을 여러 에이전트가 서로 다르게 하게 둠

### 3. 검증 관점이 다른 경우

구현 에이전트와 검증 에이전트는 분리하는 편이 좋다.

- 구현 에이전트는 "어떻게 만들지"에 집중한다.
- QA 에이전트는 "무엇이 깨질 수 있는지"에 집중한다.
- 리뷰 에이전트는 "이 변경이 너무 크거나 위험하지 않은지"를 본다.

## 생성하지 않는 게 좋은 경우

- 10분 안에 끝나는 단순 수정
- 파일 하나의 작은 문구 변경
- 요구사항이 아직 불명확한 상태
- 책임 범위가 겹쳐 충돌 가능성이 큰 작업
- 최종 판단이 사람에게 필요한 정책 결정

## 서브에이전트 요청 양식

```text
역할:
담당 범위:
읽어야 할 파일:
수정 가능한 파일:
수정 금지 파일:
출력 산출물:
검증 기준:
통합 시 주의점:
```

## 전체관리자의 역할

서브에이전트를 만들었다면 전체관리자가 반드시 해야 할 일이 있다.

- 각 에이전트의 책임 범위를 겹치지 않게 한다.
- 결과물 형식을 통일한다.
- 서로 충돌하는 판단을 확인한다.
- 최종 산출물에 반영할 것과 보류할 것을 구분한다.
- 검증 결과를 하나의 문서로 묶는다.
