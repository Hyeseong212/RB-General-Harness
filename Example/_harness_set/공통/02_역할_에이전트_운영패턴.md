# 역할 에이전트 운영 패턴

## 목적

역할 에이전트는 업무를 사람 조직처럼 나누기 위한 기준이다.

한 에이전트에게 모든 일을 맡기면 빠르게 보일 수 있지만, 결과 검증과 책임 분리가 어렵다.  
반복 가능한 업무로 만들려면 역할을 나눠야 한다.

## 기본 역할

| 역할 | 하는 일 | 산출물 |
|---|---|---|
| 전체관리자 | 목표, 범위, 우선순위, 통합 기준 관리 | Task 목록, 진행 상태, 통합 결과 |
| 분석 담당 | 입력 자료를 읽고 현재 상태를 정리 | 현황 요약, 문제 목록, 기준 자료 |
| 실행 담당 | 정해진 범위 안에서 산출물 작성 | 코드, 문서, 표, 디자인 초안 |
| 검증 담당 | 결과가 기준에 맞는지 확인 | QA 체크리스트, 리뷰 코멘트 |
| 문서화 담당 | 결과와 다음 작업을 남김 | 변경 기록, 인수인계 문서 |
| 최종 승인자 | 사람이 판단해야 하는 부분을 확정 | 승인/보류/수정 요청 |

## 언제 역할을 나누는가

- 입력 자료가 많다.
- 한 작업 안에 분석, 실행, 검증이 모두 들어 있다.
- 결과가 다른 부서로 전달된다.
- 실패했을 때 책임 범위를 알아야 한다.
- 같은 유형의 업무를 반복할 예정이다.

## 역할 분리 예시

```text
전체관리자:
- 목표와 완료 기준을 정한다.
- 산출물을 하나로 묶는다.

분석 담당:
- 기존 문서와 요구사항을 읽는다.
- 빠진 정보와 리스크를 정리한다.

실행 담당:
- 정해진 출력 형식으로 초안을 만든다.

검증 담당:
- 누락, 충돌, 과장, 위험 요소를 찾는다.

최종 승인자:
- 실제 제출/반영 여부를 결정한다.
```

## 주의점

- 역할을 너무 많이 나누면 통합 비용이 커진다.
- 역할이 겹치면 서로 다른 결론이 나온다.
- 검증 담당은 실행 담당과 분리하는 편이 좋다.
- 최종 판단이 필요한 업무는 반드시 사람 승인 단계를 둔다.
