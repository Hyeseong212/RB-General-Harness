# UI/UX 실무기획 하네스

- 작성일: 2026-05-12
- 적용 단계: third acting
- 목적: UI/UX 기획 PPT를 개발 인계용 화면정의서 형식으로 낮춘다.

## 1. 산출물 원칙

UI/UX 기획 산출물은 추상 문장이 아니라 아래 항목을 채운 표로 작성한다.

| 구분 | 반드시 적을 것 |
| --- | --- |
| 화면 | 화면ID, 화면명, 사용자, 진입 조건, 종료 조건 |
| 동작 | 버튼명, 클릭/탭 결과, 다음 화면ID |
| 입력 | 필드명, 타입, 필수 여부, 허용값, 오류 문구 |
| 상태 | 상태 코드, 진입 이벤트, 허용 버튼, 저장 이벤트 |
| 데이터 | 엔티티명, 필드명, 생성 시점, 사용 화면 |
| 예외 | 발생 조건, 화면 문구, 다음 처리 |
| 완료 조건 | 개발자가 확인할 수 있는 체크 문장 |

## 2. 화면정의서 템플릿

| 항목 | 내용 |
| --- | --- |
| 화면ID | 예: `P-03` |
| 화면명 | 예: `미션 수행` |
| 사용자 | 환자 / 치료사 / 관리자 |
| 진입 조건 | 이전 화면, 버튼, 상태 |
| 노출 데이터 | 화면에 표시되는 값 |
| 입력값 | 사용자가 입력하는 값 |
| 버튼 | 버튼명, 활성 조건, 클릭 결과 |
| 저장 이벤트 | 이벤트명, 저장 필드 |
| 오류/예외 | 발생 조건, 표시 문구, 이동 화면 |
| 완료 조건 | 테스트 가능한 문장 |

## 3. 사용자 스토리 템플릿

| 항목 | 작성 기준 |
| --- | --- |
| Actor | 실제 사용자 역할 |
| Action | 화면에서 하려는 일 |
| Goal | 그 일을 하는 이유 |
| Acceptance Criteria | `완료 조건:`으로 시작하는 테스트 문장 |

## 4. 와이어프레임 템플릿

| 항목 | 작성 기준 |
| --- | --- |
| 화면 제목 | 와이어프레임의 화면명 |
| 인터랙션 요소 | 버튼, 입력 필드, 토글, 목록 |
| 요소 설명 | 모호한 요소는 위치 옆에 설명 |
| 다음 화면 | 각 버튼이 이동시키는 화면ID |
| 상태 변화 | 클릭 후 버튼, 메시지, 저장값 변화 |

## 5. 상태/이벤트 템플릿

| 상태 | 진입 이벤트 | 허용 버튼 | 저장 이벤트 | 다음 상태 |
| --- | --- | --- | --- | --- |
| `READY` | 화면 진입 | 시작, 뒤로 | 없음 | `ACTIVE` |
| `ACTIVE` | 시작 탭 | 쉬어가기, 중단 입력 | `session_started`, `movement_tick` | `REST`, `STOPPED`, `COMPLETE` |
| `REST` | 쉬어가기 탭 | 계속, 중단 입력 | `rest_started` | `ACTIVE`, `STOPPED` |
| `STOPPED` | 중단 사유 제출 | 저장하고 종료 | `stop_reason_saved` | `COMPLETE` |
| `COMPLETE` | 완료 조건 도달 | 저장하고 나가기 | `session_completed` | 종료 |

## 6. 문구 규칙

| 사용 가능 | 사용 금지 |
| --- | --- |
| `목표 지점에 가까워지고 있어요.` | `기능이 좋아지고 있어요.` |
| `기록되었습니다.` | `회복되었습니다.` |
| `쉬어가도 됩니다.` | `계속해야 효과가 있습니다.` |
| `중단 사유를 선택해 주세요.` | `통증은 정상입니다.` |
| `치료사 확인이 필요합니다.` | `치료 효과가 확인되었습니다.` |

## 7. 개발 인계 체크

- 화면ID가 있는가
- 모든 버튼에 클릭 결과가 있는가
- 모든 입력값에 타입과 필수 여부가 있는가
- 오류/예외 문구가 있는가
- 상태 전이가 표로 정리되어 있는가
- 저장 이벤트명이 있는가
- 완료 조건이 테스트 문장으로 적혀 있는가
- 재활 전문가 확인 항목이 분리되어 있는가
