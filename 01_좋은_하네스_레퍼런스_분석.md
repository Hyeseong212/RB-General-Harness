# 좋은 하네스 레퍼런스 분석

인터넷에서 많이 쓰이는 에이전트 운영 방식들을 보면, 좋은 하네스는 단순한 프롬프트 모음이 아니라 "반복 업무를 다시 실행할 수 있게 만드는 운영 파일"에 가깝습니다.

## 참고한 공개 패턴

| 패턴 | 핵심 아이디어 | 우리 회사 적용 |
| --- | --- | --- |
| AGENTS.md | 에이전트가 먼저 읽는 저장소용 README | 프로젝트마다 `AGENTS.md` 또는 루트 운영 지침을 둔다 |
| Cursor Rules | 규칙을 파일 단위로 쪼개고 적용 범위를 정한다 | 개발, 디자인, 기획 등 폴더별 규칙을 분리한다 |
| Cline Memory Bank | 세션이 바뀌어도 프로젝트 맥락을 유지한다 | `activeContext`, `progress`, `decision-log`를 남긴다 |
| Claude Code Subagents | 역할별 에이전트를 분리한다 | 분석, 구현, 검증, 문서화 역할을 나눈다 |
| Claude Code Hooks | 특정 시점에 검증 또는 차단을 실행한다 | 배포, DB, 민감정보, 외부제출 전 승인 게이트를 둔다 |
| Aider CONVENTIONS.md | 코딩 규칙을 읽기 전용 문서로 주입한다 | 개발팀의 기술 스택, 테스트 명령, 금지 패턴을 짧게 둔다 |
| GitHub Copilot instructions | 저장소 공통 지침과 작업별 prompt file을 분리한다 | 루트 지침과 작업별 Task brief를 나눈다 |
| Windsurf Rules/Workflows/Skills | durable knowledge는 Rules/AGENTS.md에 둔다 | 기억해야 할 조직 규칙은 채팅이 아니라 MD 파일에 남긴다 |
| 공식 법령/규제 포털 | 최신 법률과 규제는 모델 기억이 아니라 원문 확인이 필요하다 | 법률, 규제, 의료기기, 개인정보는 팩트체킹 로그를 남긴다 |

## 좋은 하네스가 갖는 공통 구조

### 1. 진입점 문서

에이전트가 가장 먼저 읽는 문서입니다.

- 프로젝트 목적
- 폴더 구조
- 주요 명령
- 금지 영역
- 검증 기준
- 사람 승인 게이트

예시 파일:

- `AGENTS.md`
- `CLAUDE.md`
- `.github/copilot-instructions.md`
- `.cursor/rules/index.mdc`

### 2. 범위별 규칙

모든 규칙을 한 파일에 넣으면 에이전트가 중요한 내용을 놓치기 쉽습니다. 좋은 하네스는 적용 범위를 나눕니다.

- 전체 공통 규칙
- 개발 규칙
- 디자인 규칙
- 기획 규칙
- 테스트 규칙
- 보안 규칙
- 외부 제출 규칙

### 3. 메모리 파일

세션이 바뀌면 대화 맥락은 사라질 수 있습니다. 그래서 작업 상태를 파일로 남겨야 합니다.

추천 구조:

```text
memory-bank/
├── projectbrief.md
├── productContext.md
├── activeContext.md
├── systemPatterns.md
├── techContext.md
├── decision-log.md
└── progress.md
```

### 4. 역할 분리

한 에이전트에게 분석, 구현, 검증, 문서화를 모두 맡기면 결과가 흐려질 수 있습니다. 좋은 하네스는 역할을 나눕니다.

- Owner: 요청자, 최종 승인자
- Planner: 요구사항 분해
- Implementer: 실제 작업
- Reviewer: 검증과 리스크 확인
- Documenter: 산출물 정리

### 5. 검증 게이트

결과를 바로 확정하지 않고 업무로 넘어가기 전 검증 관문을 둡니다.

- PASS: 바로 사용 가능
- WARN: 사용할 수 있지만 사람 확인 필요
- FAIL: 다시 작업 필요
- BLOCKED: 입력 자료 부족 또는 승인 필요

### 6. 인계 문서

좋은 하네스는 마지막 답변이 아니라 다음 사람이 이어받을 수 있는 문서를 남깁니다.

필수 항목:

- 무엇을 했는가
- 어떤 파일을 만들거나 바꿨는가
- 어떻게 검증했는가
- 남은 리스크는 무엇인가
- 다음 담당자는 무엇을 보면 되는가

### 7. 팩트체킹 로그

법률, 규제, 가격, 정부지원사업, 의료기기 인허가, 개인정보처럼 바뀔 수 있는 정보는 별도 로그를 남깁니다.

필수 항목:

- 확인한 날짜
- 검색한 키워드
- 공식 출처 URL
- 문서명과 발행기관
- 시행일, 개정일, 고시일, 등록일
- 확인한 근거 문장
- 에이전트 해석
- 사람 승인 필요 여부

## 우리 회사용 결론

우리에게 필요한 하네스는 "멋진 프롬프트"가 아니라 아래 4개 세트입니다.

1. 부서별 작업 지침
2. 작업별 Task brief
3. 메모리/인계 문서
4. 사람 승인 게이트
5. 팩트체킹 로그

이 구조로 두면 개발자뿐 아니라 기획, 디자인, 하드웨어, F/W, 회로, 경영지원, 재활 도메인까지 같은 방식으로 에이전트를 운영할 수 있습니다.
