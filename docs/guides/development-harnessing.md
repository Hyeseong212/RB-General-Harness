# 개발 하네싱 가이드

## 목적

개발 에이전트는 코드를 대신 책임지는 존재가 아니라, 분석, 구현 초안, 테스트, 문서화를 빠르게 묶어주는 작업 보조자다.

## 권장 역할

| 에이전트 | 역할 |
|---|---|
| 레거시 분석 에이전트 | 기존 코드, DB, API, 화면 동작을 조사한다. |
| 구현 에이전트 | 명확한 범위 안에서 코드 변경안을 만든다. |
| 코드리뷰 에이전트 | 버그, 회귀, 보안, 누락 테스트를 찾는다. |
| QA 에이전트 | 빌드, API, UI, 권한, 데이터 범위 테스트를 수행한다. |
| 문서화 에이전트 | 변경 이유, 사용법, 남은 리스크를 문서로 남긴다. |

## 개발 Task 입력

개발 Task를 맡기기 전에 아래를 적는다.

- 목표 기능
- 관련 화면 또는 API
- 기준이 되는 기존 동작
- 수정 가능한 파일 범위
- 건드리면 안 되는 영역
- 완료 기준
- 테스트 방법

## 좋은 Task 예시

```text
환자 상세 화면의 저장 동작을 OFBiz 기준과 맞춘다.
입력: 기존 Vue 화면, Spring API, OFBiz 저장 흐름, DB 테이블 목록
출력: 수정 코드, API 변경점, 검증 결과, 남은 리스크
검증: npm build, Maven compile, 환자 상세 저장 후 새로고침 확인
```

## 검증 기준

- 빌드가 통과했는가
- 기존 기능을 깨지 않았는가
- API 응답 형식이 유지되는가
- 권한별 데이터 범위가 맞는가
- 운영 DB나 민감 데이터에 영향을 주지 않았는가
- 변경 이유와 남은 리스크가 문서로 남았는가

## 금지 패턴

- "전체 코드 리팩토링해줘"처럼 범위가 넓은 요청
- 테스트 없이 결과만 받는 방식
- 에이전트가 만든 코드를 사람이 읽지 않고 바로 반영
- 레거시 기준 동작을 확인하지 않고 화면만 새로 만드는 방식
