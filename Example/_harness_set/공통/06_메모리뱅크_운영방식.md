# 메모리뱅크 운영 방식

에이전트 대화는 길어질수록 맥락이 흐려질 수 있습니다. 메모리뱅크는 프로젝트의 중요한 맥락을 대화가 아니라 파일로 남기는 방식입니다.

## 왜 필요한가

- 새 대화에서도 이전 결정과 진행 상태를 이어갈 수 있다.
- 사람이 바뀌어도 같은 기준으로 작업할 수 있다.
- 발표자료, 개발, 디자인, 구매, 재활 검토가 같은 프로젝트 맥락을 공유할 수 있다.
- "전에 말했던 것"을 다시 설명하는 시간을 줄인다.

## 추천 폴더 구조

```text
memory-bank/
├── projectbrief.md
├── productContext.md
├── activeContext.md
├── systemPatterns.md
├── techContext.md
├── decision-log.md
├── progress.md
└── risk-register.md
```

## 파일별 역할

| 파일 | 내용 | 업데이트 시점 |
| --- | --- | --- |
| `projectbrief.md` | 프로젝트 목적, 범위, 성공 기준 | 프로젝트 시작 또는 범위 변경 |
| `productContext.md` | 사용 대상, 문제, UX 목표 | 제품 방향 변경 |
| `activeContext.md` | 지금 집중 중인 작업, 최근 변경 | 매 작업 시작/종료 |
| `systemPatterns.md` | 구조, 반복 패턴, 설계 원칙 | 구조 변경 |
| `techContext.md` | 기술 스택, 실행 명령, 제약 | 개발 환경 변경 |
| `decision-log.md` | 결정 사항과 이유 | 중요한 판단 직후 |
| `progress.md` | 완료, 진행 중, 남은 일 | 주간 또는 작업 종료 |
| `risk-register.md` | 리스크, 담당자, 대응 상태 | 리스크 발견 시 |

## 에이전트에게 주는 시작 문구

```md
작업을 시작하기 전에 memory-bank/projectbrief.md, activeContext.md, progress.md를 먼저 읽어라.
이후 현재 요청과 관련된 파일만 추가로 확인하고, 작업 종료 시 activeContext.md와 progress.md에 변경 요약을 남겨라.
```

## 사용 규칙

- 모든 메모리를 매번 읽게 하지 않는다.
- 시작 시에는 `projectbrief`, `activeContext`, `progress`만 기본으로 읽는다.
- 필요할 때 `techContext`, `decision-log`, `risk-register`를 추가로 읽는다.
- 오래된 내용은 삭제보다 "대체됨"으로 표시한다.
- 민감 정보는 메모리뱅크에 원문으로 넣지 않는다.

## 우리 회사 적용 예

### 개발

- 현재 리팩토링 범위
- 테스트 통과 여부
- 건드리면 안 되는 운영 파일
- 다음 개발자가 볼 인계 사항

### 기획

- 과제 제출 방향
- PPT 핵심 메시지
- 외부 제출 전 승인 필요 문구
- 회의 결정 사항

### 디자인

- PPT 스타일 기준
- 캡처 이미지 사용 여부
- 마스킹 필요한 화면
- 시각 자료 검토 결과

### 하드웨어/FW/회로

- 장비 버전
- 펌웨어 버전
- 변경된 핀맵 또는 프로토콜
- 실제 장비 테스트 필요 항목

## 종료 시 인계 규칙

작업 종료 시 아래 5줄은 반드시 남깁니다.

```md
## 이번 작업
## 변경 파일
## 검증 결과
## 남은 리스크
## 다음 액션
```
